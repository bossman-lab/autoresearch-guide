## 八、完整编排逻辑

整合 Fast Path 和 Slow Path 的完整流程：

```python
def auto_research_dispatch(plan_path, issue_number):
    """
    Fast/Slow 双路径编排器
    """
    # 1. 读取计划
    plan = read_plan(plan_path)
    tasks = parse_tasks(plan)
    
    # 2. 评估复杂度（是否直接走 Slow）
    complexity_score = assess_complexity(plan)
    program_md = load_program_md()
    
    for task in tasks:
        task_result = None
        
        # 3. 决策：Fast or Slow
        if complexity_score >= 8 or "security" in task.get("tags", []):
            # 直接走 Slow Path
            task_result = slow_path(task, program_md)
        else:
            # 走 Fast Path
            task_result = fast_path(task)
            
            # Fast 不通过？判断是否升级
            if task_result["status"] == "failed":
                if should_upgrade(task, task_result["review"], task_result.get("fail_count", 0)):
                    print(f"⏫ 升级到 Slow Path: {task['description'][:40]}...")
                    task_result = slow_path(task, program_md)
        
        # 4. 处理结果
        handle_result(task_result, issue_number)
        
        # 5. 日志
        log_to_tsv(task, task_result, issue_number)
```

### 9.3 风险分级审核——解决 AI 带来的 Review 瓶颈

Faros Paradox（2026 年工程报告）揭示了一个残酷事实：**AI 让代码生成速度提升 3-5 倍，但 PR 审核时间延长了 91%**。如果不解决审核瓶颈，Fast Path 的"快"毫无意义。

**方案**：对每个变更自动评分风险级别，分级处理：

```python
def risk_score(pr):
    score = 0
    score += len(pr.files_changed) * 0.5          # 改动文件越多分越高
    score += sum(1 for f in pr.files if "security" in f or "auth" in f) * 3  # 安全文件加权
    score += pr.insertions / 100                   # 代码量
    score += len(pr.dependency_changes) * 5        # 依赖变更高风险
    return score

risk = risk_score(pr)
if risk < 5:
    auto_merge = True      # 🟢 低风险 → Fast Path → 自动合并
elif risk < 15:
    need_review = True     # 🟡 中风险 → Fast Path + 人工抽查
else:
    slow_path = True       # 🔴 高风险 → 直接走 Slow Path
```

**CI 中的风险评分面板：**

| 评分维度 | 低风险（🟢） | 中风险（🟡） | 高风险（🔴） |
|---------|:----------:|:----------:|:----------:|
| 文件改动 | 1-2 个文件 | 3-5 个文件 | 6+ 个文件 |
| 安全影响 | 无安全相关 | 涉及非关键数据 | 涉及用户数据/支付 |
| 依赖变更 | 无 | 小版本升级 | 新依赖/大版本 |
| 逻辑复杂度 | 单文件内部修改 | 跨模块调用 | 架构性变更 |
| **处理方式** | Fast Path + 自动合并 | Fast Path + 人工抽查 | Slow Path + 交叉审核 |

这解决了 Faros Paradox：不是让人力审核追上 AI 生成速度，而是**让 AI 审核承担 80% 的评审工作，只把最关键的 20% 留给人工**。这正是我们 Fast/Slow 分级思想的自然延伸。

## 📖 故事：跳过筑基的代价——一家公司的教训

### 背景

某中型互联网公司（200+ 研发人员）在 2025 年 Q4 决定推行 Fast/Slow 双路径系统。技术 VP 在 KubeCon 上听到相关概念后热血沸腾，要求团队"两周内全面上线"。

### 他们做了什么

团队跳过"筑基"阶段，直接进入"试点"：

1. **没有做 Workshop** — 只在钉钉群发了一篇文档
2. **没有达成 Program.md 共识** — 架构师自己写了 Program.md，未经过团队 Review
3. **没有建沙盒环境** — 直接在线上仓库的目标分支上跑
4. **没有设定评分校准环节** — 9.0 分阈值直接沿用，没有人验证 AI 评分是否合理

Week 1 的数据：

| 指标 | 期望值 | 实际值 |
|:----:|:-----:|:-----:|
| Fast Path 执行任务数 | 10+ | 32 |
| Fast Path 成功率 | 70-85% | **43%** |
| Fast→Slow 升级率 | 15-30% | **57%** |
| 代码被回滚率 | <5% | **22%** |
| 团队信任度（匿名问卷） | >70% | **28%** |

### 发生了什么

跳过筑基阶段导致了三个连锁问题：

**问题一：Program.md 成了"空中楼阁"**。架构师制定的评分标准中，安全维度权重只有 10%，但团队中有安全背景的资深工程师指出，该公司的支付模块占总代码量的 30%。误把"对所有代码通用"的标准用在"有强安全约束的代码"上，导致 Fast Path 频繁通过本应升级的安全变更。

**问题二：没有校准的评分成了"瞎打分"**。由于没有筑基阶段的人工审核校准，AI 评分的偏差未被发现。抽样显示：AI 给"表面工整但逻辑有漏洞"的代码平均打了 8.7 分，而人工审核平均只给 5.3 分——偏差高达 3.4 分。

**问题三：团队出现"AI 盲信"与"AI 抵触"两极分化**。初级工程师完全信任 AI 评分结果，直接合并了得分 9.2 但存在并发 bug 的代码，导致线上事故。资深工程师则完全排斥系统，认为"AI 根本不懂我们的业务逻辑"。

### 修复与转机

第 3 周，技术 VP 叫停全线推行，退回筑基阶段：

1. **补办 Workshop** — 用回滚事故作为案例，全员讨论 Program.md 应该长什么样
2. **重新校准** — 组织 3 轮人工 vs AI 评分对标会，发现并修正了 7 类系统性偏差
3. **沙盒试跑** — 在隔离环境跑完 20 个任务后才回到线上
4. **分层信任** — 前 2 周 AI 评分仅作"建议"，人工必须复查；第 3 周起逐步放权

筑基补课后，数据恢复正常：

| 指标 | 筑基前 | 筑基后 |
|:----:|:-----:|:-----:|
| Fast Path 成功率 | 43% | 78% |
| Fast→Slow 升级率 | 57% | 22% |
| 代码回滚率 | 22% | 3% |
| 团队信任度 | 28% | 74% |

**核心教训**：筑基不是"走形式"，它是校准信任标尺的过程。跳过筑基直接跑 Fast Path，就像没测视力就上跑道——要么撞墙，要么跑偏。

## ⚠️ 失败案例：9.0 阈值设定失误——当 90% 的任务都走了 Slow Path

### 场景

某金融科技团队（50 人研发）在落地 Fast/Slow 方案时，为了避免风险，将评分阈值设定为 **9.5/10**（高于默认的 9.0），期望"只有最高质量的代码才能走 Fast Path"。

### 数据表现

首周运行数据（共 47 个任务）：

| 评分区间 | 任务数 | 占比 | 处理路径 |
|:--------:|:-----:|:---:|:-------:|
| 9.5 - 10 | 2 | 4% | Fast Path（自动通过） |
| 9.0 - 9.4 | 8 | 17% | ❌ 应走 Fast 但被阈值挡住 → Slow Path |
| 8.0 - 8.9 | 21 | 45% | Slow Path |
| < 8.0 | 16 | 34% | Slow Path |

**结果**：仅 4% 的任务走了 Fast Path，96% 走了 Slow Path。Fast Path 形同虚设，团队反而增加了工作量——既要跑 AI 的 Slow Path 审核，又要等人工复查。

### 根因分析

1. **阈值缺乏校准** — 9.5 的设定是"拍脑袋"决定的，没有任何数据支撑。团队没有先跑一段无阈值限制的观察期来了解评分分布。

2. **Program.md 评分标准与任务类型不匹配** — 该团队主要做内部工具库的维护和文档生成，这类任务天然得分高（简单 + 测试覆盖容易），但金融业务的合规代码又天然得分低（安全约束多 + 变更影响大）。用统一阈值忽略了这个差异。

3. **没有区分"质量门槛"和"路径选择门槛"** — 9.5 作为"代码能否上线"的质量门槛是合理的，但作为"走 Fast 还是 Slow"的路径选择门槛太高。路径选择门槛应关注"风险有多大"，而非"质量有多高"。

### 调整方案

团队在第 3 周重新设计决策逻辑：

```python
# 错误做法：单一阈值决定一切
if score >= 9.5:   # ← 这是质量门槛，不是路径选择门槛
    fast_path()

# 正确做法：双维度决策
def path_decision(score, risk_tags, complexity):
    # 路径选择：基于风险，而非质量
    if risk_tags or complexity >= 8:
        slow_path()            # 高风险 → 强制 Slow
    elif score >= 7.5:
        fast_path()            # 低风险且质量达标 → Fast
    else:
        slow_path()            # 低风险但质量不足 → 回炉重造

# 质量门槛单独保留，在合并前检查
def merge_gate(score, has_human_review):
    if score < 9.0 and not has_human_review:
        block_merge("评分不足 9.0，需人工审核后方可合并")
```

**关键区别**：
- **路径选择门槛**（Fast vs Slow）= 7.5 分，关注风险
- **合并放行门槛**（能否上线）= 9.0 分，关注质量
- Fast Path 只是"AI 全自动处理"的路径，不代表"免审上线"
- Fast Path 通过的代码仍需经过合并质量门禁

### 调整后的数据

| 指标 | 调整前（9.5 单一阈值） | 调整后（双维度决策） |
|:----:|:--------------------:|:------------------:|
| Fast Path 占比 | 4% | 71% |
| Slow Path 占比 | 96% | 29% |
| 人工介入率 | 100%（等审核慢） | 29%（仅高风险/低质量） |
| 平均 PR 合并周期 | 4.7 小时 | 22 分钟 |
| 线上缺陷发现率 | 0.8% | 0.9%（无显著变化） |

**核心教训**：阈值设定要有数据支撑。先无阈值跑 2 周看分布，再根据实际分布设定合理切分点。路径选择看风险，质量门禁看分数——不要把两个门槛混在一起。

> **张工**：那这个 Fast/Slow 是干啥的？不还是 AI 在审吗？

### 场景三：引导初级工程师 — 回答"让 AI 帮我写代码，不是爽死了？"

**背景**：初级工程师小李，入职半年，对 AI 充满热情，但缺乏判断力。他准备把 Fast Path 当作"免审核快速上线通道"。

**你的话术（思考 2 分钟，写下要点）：**

↓

**参考对话脚本**：

> **你**：小李，我看你最近试了 Fast Path，感觉怎么样？
>
> **小李**：太爽了！写个工具函数，AI 帮我审完就直接合并了，不用等人 Review，效率翻倍！
>
> **你**：嗯，效率确实是提升了。不过我想问你一个问题——上次你那个工具函数，AI 评分打了 9.2 分，你知道它扣分扣在哪里吗？
>
> **小李**：呃……没仔细看，反正是通过了。
>
> **你**：我看了那个报告，0.8 分扣在安全维度——是因为函数接受用户输入但没有做长度校验。9.2 分确实通过了 Fast Path，但这不代表这代码可以直接上线。它只是说明"这个变更风险低，AI 可以代为初审"。合并之前，质量门禁会检查安全相关的警告。

> **你**：给你三个简单判断标准：
>
> 1. **如果你自己都不确定代码对不对** → 主动要求走 Slow Path
> 2. **如果改动的模块你只读过没写过** → 主动要求走 Slow Path
> 3. **如果你完全理解改动内容，且不涉及业务逻辑** → 放心走 Fast Path
>
> **记住这个口诀**：`Fast 跑得快，Slow 保得稳；不确定就升级，不丢人。`
