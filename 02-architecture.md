> **一句话摘要**：本文详细设计一套 Fast/Slow 双路径架构，让 AI 辅助开发在"快"和"准"之间自动做出最优选择，成本和质量的动态平衡。

---

## 一、问题定义

### 1.1 两个相互矛盾的目标

在 AI 辅助开发的工程实践中，我们面临一对根本矛盾：

| 目标 | 要求 | 代价 |
|------|------|------|
| **快** | 减少审核轮次，一次交付 | 简单任务省心，复杂任务容易翻车 |
| **准** | 反复验证，多轮迭代 | 复杂任务可靠，简单任务浪费资源 |

**关键洞察**：这两个目标不冲突，前提是——系统能自动区分当前任务应该走哪条路。

### 1.2 "一刀切"的问题

| 方案 | 对简单任务 | 对复杂任务 |
|------|-----------|-----------|
| 全走 Fast Path | ✅ 省 token，快 | ❌ 质量不可控，翻车率高 |
| 全走 Slow Path | ❌ 严重浪费 | ✅ 质量有保障 |
| 人工分类 | ❌ 人需要判断，不精确 | ❌ 人需要判断，不精确 |

**解决方案**：让系统自动分级，根据执行结果动态升级。

## 二、Fast/Slow 双路径架构设计

### 2.1 架构概览

```
┌─────────────────────────────────────────────────────────┐
│                   任务调度器 (Orchestrator)               │
│  • 接收任务 → 解析复杂度 → 决定初始路径                    │
│  • 路径降级/升级决策                                      │
│  • 全量日志存档                                          │
└────────────────────┬────────────────────────────────────┘
                     │
         ┌───────────┴───────────┐
         ▼                       ▼
┌──────────────────┐   ┌──────────────────────────────┐
│   Fast Path      │   │        Slow Path              │
│                  │   │                              │
│ ┌──────────────┐ │   │ ┌──────────────────────────┐ │
│ │ Step 1       │ │   │ │ Step 1: 加载 program.md  │ │
│ │ 任务分解     │ │   │ │（宪法规则注入）            │ │
│ └──────┬───────┘ │   │ └──────────┬───────────────┘ │
│        ▼         │   │            ▼                  │
│ ┌──────────────┐ │   │ ┌──────────────────────────┐ │
│ │ Step 2       │ │   │ │ Step 2: 实施周期循环      │ │
│ │ 实现 + 测试  │ │   │ │ ┌──────────────────┐    │ │
│ └──────┬───────┘ │   │ │ │ 实施者 A 实现    │    │ │
│        ▼         │   │ │ └────────┬─────────┘    │ │
│ ┌──────────────┐ │   │ │          ▼               │ │
│ │ Step 3       │ │   │ │ ┌──────────────────┐    │ │
│ │ 快速审核     │ │   │ │ │ 实施者 B 审核 →  │    │ │
│ │（二元过/不过）│ │   │ │ │ 评分 + 问题列表  │    │ │
│ └──────┬───────┘ │   │ │ └────────┬─────────┘    │ │
│        ▼         │   │ │          ▼               │ │
│   ┌────────┐     │   │ │ ┌──────────────────┐    │ │
│   │ 通过   │     │   │ │ │ 达标？ ≥ 9.0     │────┼────→ ✅
│   │ 不通过 │─────┼───┼─┼─┤ 否→反馈注入下一轮│    │ │
│   └────────┘     │   │ │ │ 达上限？→人工介入│    │ │
│        │         │   │ │ └──────────────────┘    │ │
│        ▼         │   │ └──────────────────────────┘ │
│      ✅ 交付      │   └──────────────────────────────┘
└──────────────────┘
```

### 2.2 核心组件说明

#### 2.2.1 任务调度器（Orchestrator）

职责：
- 接收任务输入
- 从 plan 中解析任务复杂度（基于文件数、跨模块边界数、关键性标记）
- 决定初始路径（默认 Fast，但满足条件时直接走 Slow）
- 跟踪执行状态，触发路径升级/降级

**复杂度评估规则**：

```python
def assess_complexity(task):
    score = 0
    score += task.files_changed * 2          # 每个文件 +2
    score += task.cross_module * 5            # 跨模块 +5
    score += 3 if "security" in task.tags else 0  # 安全标签 +3
    score += 3 if "breaking" in task.tags else 0   # 破坏性变更 +3
    score += 1 if task.estimated_hours > 2 else 0  # 预估 >2h +1
    
    if score >= 8:
        return "slow"    # 复杂度高 → 直接走 Slow Path
    return "fast"        # 复杂度低 → 先走 Fast Path
```

#### 2.2.2 Fast Path 引擎

职责：
- 将一个子任务交给单个实施 agent
- 实施 agent 执行 TDD 流程（测试先行 → 实现 → 验证）
- 快速二元审核（正确性 + 关键路径测试）
- 判断是否升级到 Slow Path

**升级触发条件**（任一满足即升级）：
1. 审核不通过，且是**跨模块**或**高安全**任务
2. 连续 2 次 Fast Path 执行失败
3. Fast Path 审核发现的问题**无法在小幅度修改内解决**
4. 实施 agent 主动报告"这个任务可能需要反复调整"

#### 2.2.3 Slow Path 引擎

职责：
- 加载 program.md 宪法规则
- 双模型（A ↔ B）交叉实施 + 审核
- 5 维量化评分
- 反馈驱动迭代

**迭代引擎伪代码**：

```python
def slow_path(task, program_md):
    max_rounds = 42
    scores = []
    
    for round in range(1, max_rounds + 1):
        # 轮换角色
        if round % 2 == 1:
            implementer = model_A
            reviewer = model_B
        else:
            implementer = model_B
            reviewer = model_A
        
        # 实施
        code = implementer.run(task, context=f"""
            上一轮反馈：{get_previous_feedback(round, scores)}
            规则：{program_md}
        """)
        
        # 测试
        test_result = run_tests(code)
        if not test_result.passed:
            scores.append({"round": round, "passed": False})
            continue
        
        # 审核 + 评分
        review = reviewer.review(code, dimensions=SCORE_DIMENSIONS)
        scores.append(review)
        
        # 判断
        if review.weighted_total >= 9.0:
            return {"status": "approved", "round": round, "code": code}
        
        # 评分未提高？判断
        if round >= 3 and not is_improving(scores[-3:]):
            return {"status": "escalate", "reason": "评分停滞"}
    
    return {"status": "escalate", "reason": "超过最大轮次"}
```

## 三、关键设计决策

### 3.1 如何判断 Fast Path 失败？

**不是"一次不通过"就升级**。正确的判断逻辑：

| 条件 | 行为 |
|------|------|
| 审核不通过 + 问题**可小幅度修复** | 让实施 agent 修复后重审（仍在 Fast Path 内） |
| 审核不通过 + 问题**涉及架构层面** | 立即升级到 Slow Path |
| 实施 agent **自己说"需要多次迭代"** | 尊重 agent 的判断，升级 |
| 连续 2 次修复后审核仍不通过 | 升级 |

### 3.2 评分阈值为什么是 9.0？

这是"够好但不高到浪费"的平衡点。

| 阈值 | 含义 | 达成率（估算） | 平均迭代轮次 |
|------|------|---------------|-------------|
| 7.0 | 能用 | ~95% | 1-2 |
| 8.0 | 良好 | ~85% | 2-3 |
| **9.0** | **优秀** | **~70%** | **3-5** |
| 9.5 | 卓越 | ~40% | 6-10 |
| 10.0 | 完美 | ~5% | 15+ |

9.0 意味着：**不是所有方面都完美，但没有致命缺陷，主要的改进方向明确。** 超过 9.5 的边际成本太高，大多数业务场景不需要。

### 3.3 怎么避免死循环？

设置三类终止条件：

```
终止 A — 正常退出：评分 ≥ 9.0
终止 B — 硬性上限：最大 42 轮（约 3-4 小时）
终止 C — 动态兜底：连续 3 轮评分未提高
```

当触发终止 B 或 C 时，**不丢脸，不背锅**。系统记录全量日志（每轮的代码、评分、审核意见），交给人工判断。

## 四、工程实现要点

### 4.1 评分持久化

每次执行的结果应该持久化到 JSON 文件，用于趋势分析：

```json
{
  "task_id": "issue-21",
  "timestamp": "2026-05-18T08:00:00Z",
  "path_taken": "fast→slow",
  "fast_path": {
    "rounds": 1,
    "result": "failed",
    "reason": "跨模块依赖未正确处理"
  },
  "slow_path": {
    "total_rounds": 3,
    "final_score": 9.0,
    "score_history": [
      {"round": 1, "score": 5.0, "model_implementer": "deepseek", "model_reviewer": "claude"},
      {"round": 2, "score": 7.0, "model_implementer": "claude", "model_reviewer": "deepseek"},
      {"round": 3, "score": 9.0, "model_implementer": "deepseek", "model_reviewer": "claude"}
    ]
  }
}
```

### 4.2 通知机制

| 事件 | 通知对象 |
|------|---------|
| Fast Path 升级到 Slow | 记录日志（人工不介入） |
| Slow Path 评分达标自动合并 | 团队频道通知（含评分报告） |
| Slow Path 触发终止 B/C | **@开发负责人** 强制人工介入 |

### 4.3 降级策略（反向升级）

如果一个任务：
1. 人工评估复杂度高 → 走 Slow Path
2. 但第一轮就拿到 9.5+ → 可以提前结束

但注意：**不走从 Slow 降级到 Fast**。Slow Path 多出来的审核是质量保障，不是浪费。

## 五、与现有工作流的集成

### 5.1 与 PR 工作流结合

```
开发者创建 PR
    │
    ▼
Fast/Slow 系统自动分析变更
    │
    ├── 简单变更（增删字段、小修小补）
    │   └── Fast Path → 自动审核 → 标注 Approved
    │
    └── 复杂变更（新功能、架构调整）
        └── Slow Path → 自动审核 + 评分报告 → 标注 Approved with Report
```

评审人员在 PR 中看到的不是冰冷的"通过/不通过"，而是：

```markdown
## 🤖 AI 评审报告

**执行路径**: Slow Path（3轮迭代）

**最终评分**: 9.0 / 10 ✅

| 维度 | 评分 | 评估 |
|------|------|------|
| 正确性 | 10 | 全部用例通过 |
| 测试 | 9 | 覆盖率 87%，含边界 case |
| 代码质量 | 9 | 拆分合理，命名清晰 |
| 安全 | 8 | ⚠️ 建议对传入参数加长度校验 |
| 性能 | 9 | 无性能隐患 |

**人工关注点**：
- 参数长度校验（Security 8分）
- 如需修改请通知 @owner
```

### 5.2 与 CI/CD 集成

```
push / PR
    │
    ▼
CI pipeline 启动
    │
    ├── lint / test（常规）
    │
    └── AI 质量评分（非阻塞）
        ├── Fast Path → 直接在 CI 输出评分
        └── Slow Path → 异步执行，结果通过 webhook 推送
```
