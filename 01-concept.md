> **一句话摘要**：本文将 Karpathy 的 AutoResearch 方法论迁移到软件开发领域，提出"Fast/Slow 双路径"分级兜底策略，在不牺牲效率的前提下为代码质量建立量化保障体系。

## 一、背景：为什么需要 AutoResearch 方法论？

### 1.1 AI 辅助开发的三个时代

- **V1.0 — 对话式辅助（2023-2024）**：Copilot/ChatGPT 回答问题、补全代码。开发者写 prompt → 看结果 → 手动调整。效率提升 20%-30%，质量完全依赖人的判断。

- **V2.0 — 单 Agent 自动化（2024-2025）**：Devin、OpenClaw、Claude Code 等工具出现。单个 Agent 可以完成端到端任务。但问题是：**一个模型既当运动员又当裁判**。Agent 自己的盲区就是整个流程的盲区。

- **V3.0 — 多 Agent 对抗协作（2026+）**：AutoResearch 的开创性在于——用两个不同的模型轮流担任实现者和审核者。Codex 和 Claude 有不同的训练数据和能力侧重，它们的盲区**不重叠**。这就是"交叉审核"能发现单 Agent 发现不了的问题的根本原因。

### 1.2 为什么需要量化评分？

传统 code review 的痛点是"二元判断"——通过或不通过。但软件质量从来不是非黑即白的。

一个实现可能：
- **正确性 10/10**：功能完全正确
- **测试覆盖 7/10**：主流程覆盖了，但边界情况遗漏
- **代码质量 6/10**：可读性一般，有魔法数字
- **安全 8/10**：无明显漏洞
- **性能 9/10**：效率尚可

总分 = 10×0.35 + 7×0.25 + 6×0.20 + 8×0.10 + 9×0.10 = **8.0/10**

这个 8.0 比简单的"不通过"多了一整层信息：**问题主要出在测试覆盖和代码质量上**。下一轮的 Agent 就能有针对性地改进，而不是盲目重试。

## 二、核心策略：Fast/Slow 双路径分级兜底

### 2.1 设计理念

我们的核心洞察是：**80% 的日常开发任务是简单任务**（增删字段、改配置、修简单 bug），不值得每次都启动完整的 AutoResearch 循环。但如果这些简单任务偶尔翻车，需要一个自动升级机制来兜底。

这个思路可以用一个比喻来理解：
- **急诊分诊**：病人来了先快速分诊（Fast Path），轻症直接处理
- **专科会诊**：重症或分诊无法确诊的，启动多学科会诊（Slow Path）

### 2.2 系统架构

```
任务进入
    │
    ▼
[Fast Path — 默认路径]
    ├── Subagent 实现
    ├── 自动测试
    └── 快速审核（二元判断）
         │
    ┌────┴────────────┐
    ▼                 ▼
  通过              不通过 / 连续失败 × N
    │                 │
    ▼                 ▼
  ✅ 合并      [Slow Path — 自动升级]
                  ├── program.md 规则加载
                  ├── 双模型交叉实施
                  │   （Codex/Claude 轮流写+审）
                  ├── 5维量化评分
                  ├── 反馈驱动迭代
                  └── 全量日志存档
                       │
                  ┌────┴────────────┐
                  ▼                 ▼
               评分 ≥ 9.0         评分不达标
                  │                 │
                  ▼                 ▼
              ✅ 合并           ⛑️ 人工介入
```

### 2.3 分档矩阵

| 维度 | Fast Path | Slow Path |
|------|-----------|-----------|
| **适用场景** | 简单 bug fix、字段修改、配置变更、单文件改动 | 跨模块 feature、架构改动、高安全敏感、新系统模块 |
| **Agent 数量** | 1（实现+自审） | 2+（交叉实施+交叉审核） |
| **质量评估** | 二元通过/不通过 | 5维加权评分 ≥ 9.0 |
| **迭代机制** | 无（一次通过） | 反馈驱动循环（最多 42 轮） |
| **典型耗时** | 2-5 分钟 | 10-30 分钟 |
| **Token 消耗** | 1x | 3-8x |
| **成功率** | ~80%（简单任务） | ~95%+（经过迭代） |

## 三、5维评分体系详解

### 3.1 维度与权重

| 维度 | 权重 | 评估内容 | 高分的标准 | 低分的典型表现 |
|------|------|---------|-----------|---------------|
| **正确性** | 35% | 功能是否完全正确 | 所有用例均通过，无逻辑错误 | 主流程跑通但边界 case 失败 |
| **测试** | 25% | 测试覆盖 ≥70% | 表格驱动测试，覆盖正常+异常 | 只测 happy path，无边界 case |
| **代码质量** | 20% | 规范、可维护 | 命名清晰、结构合理、无魔法数字 | 长函数、重复代码、命名模糊 |
| **安全** | 10% | 无安全漏洞 | 输入校验、SQL 注入防护、敏感信息处理 | 未做输入过滤、硬编码密钥 |
| **性能** | 10% | 无性能隐患 | 时间复杂度合理、无内存泄漏 | N+1 查询、不必要的内存分配 |

### 3.2 评分规则

| 分数 | 含义 |
|------|------|
| 10 | 完美，无任何问题 |
| 9 | 有建议改进项，但非必要 |
| 7 | 一般问题，需要修复 |
| 4 | 严重问题，必须修复后才能合并 |
| 1 | 致命问题，设计完全错误 |

### 3.3 加权计算示例

```
正确性: 9 × 0.35 = 3.15
测试:    7 × 0.25 = 1.75
质量:    8 × 0.20 = 1.60
安全:    8 × 0.10 = 0.80
性能:   10 × 0.10 = 1.00
────────────────────
总分:              8.30 / 10
```

**决策**：低于 9.0 → 进入下一轮迭代，审核反馈注入下一轮 prompt。

## 四、program.md：Agent 的宪法

### 4.1 为什么需要 Program.md？

在 Karpathy 的 AutoResearch 中，`program.md` 是所有 Agent 的行为准绳。这个文件：

1. **消除歧义**：每个 Agent 都读到同一套规则，不会出现"我以为可以"的情况
2. **权限边界**：明确什么能做什么不能做，防止 Agent 越界
3. **质量标准**：统一的代码规范、测试标准、安全要求
4. **可审计**：所有决策的规则依据可追溯

### 4.2 标准 Program.md 模板

```markdown
# Program.md — 开发宪法

## 权限边界

### 允许
- 修改 `src/`, `cmd/`, `internal/` 目录下的代码
- 创建和修改测试文件 `tests/`, `*_test.go`
- 运行 `go test ./...`, `npm test`, `pytest` 等
- 创建 git commit（由编排引擎统一 push）

### 禁止
- 修改 CI/CD 配置 (`.github/`, `Jenkinsfile`, `Dockerfile`)
- 修改 `go.mod`, `package.json` 主版本依赖
- 删除现有文件（除非 Issue 明确要求）
- 修改 Program.md 本身
- 直接推送到远程仓库

## 代码规范

1. 遵循 Effective Go / PEP 8 / 社区约定
2. 函数长度 ≤ 50 行，文件 ≤ 500 行
3. 所有 public 函数/方法必须有文档注释
4. 禁止魔法数字（提取为 const/enum）
5. 错误处理不能吞错误（禁止 `_` 忽略 error）

## 测试规范

1. 覆盖率 ≥ 70%
2. 表格驱动测试（Table-Driven Tests）
3. 命名：`Test<Function>_<Scenario>`
4. 禁止：`time.Sleep`、外部 HTTP 依赖、全局状态

## 评分维度

| 维度 | 权重 | 最低分 |
|------|------|--------|
| 正确性 | 35% | 7 |
| 测试 | 25% | 7 |
| 代码质量 | 20% | 7 |
| 安全 | 10% | 7 |
| 性能 | 10% | 7 |

达标线：加权总分 ≥ 9.0
```

## 五、多模型交叉审核的工程原理

### 5.1 为什么不同模型交叉有效？

核心原因：**每个模型都有自己"擅长"和"不擅长"的领域**。

| 模型特性 | Codex / GPT 系 | Claude 系 |
|---------|---------------|-----------|
| **训练数据侧重** | 更大的代码语料占比 | 更注重推理和逻辑一致性 |
| **编码风格** | 更倾向于生产级代码模式 | 更注重可读性和注释 |
| **盲区典型表现** | 偶尔跳过安全检查 | 可能在性能优化上不够激进 |
| **审核特长** | 发现 API 用法错误 | 发现逻辑漏洞和边界条件 |

当 Codex 实现、Claude 审核时：Claude 检查 Codex 可能漏掉的逻辑问题  
当 Claude 实现、Codex 审核时：Codex 检查 Claude 可能忽略的工程细节

**这就是"交叉覆盖"消除盲区的数学原理。** 两个模型的错误分布是独立的，交集（两者都漏掉的问题）显著小于并集（各自漏掉的问题）。

### 5.2 轮换策略

```
轮次 1: Codex 实现 → Claude 审核 → 修复 → 评分
轮次 2: Claude 实现 → Codex 审核 → 修复 → 评分
轮次 3: Codex 实现 → Claude 审核 → 修复 → 评分
...
```

奇数轮和偶数轮互换角色，确保双方都有机会从各自的角度审视代码。

## 六、反馈驱动迭代的工程实现

### 6.1 迭代循环

```
[审核结果] → [解析评分和问题列表] → [注入下一轮 prompt] → [再次实施]
     ↑                                                    │
     └────────────────── 迭代至达标 ──────────────────────┘
```

### 6.2 下一轮 Prompt 的注入格式

```
[上一轮评审结果]

评分: 8.3 / 10
达标: 否（门槛 9.0）

各维度评分：
  - 正确性: 9 → ✅ OK
  - 测试:    7 → ❌ 提升到 ≥ 8（最薄弱项）
  - 代码质量: 8 → ⚠️ 提升到 ≥ 9
  - 安全:    8 → ⚠️ 提升到 ≥ 9
  - 性能:   10 → ✅ OK

具体问题列表（需全部修复）：
1. [测试 - 严重] 表格驱动测试只覆盖了正常路径，缺少边界 case：
   空数组输入未测试
2. [质量 - 重要] `processData()` 函数 80 行，建议拆分为 3 个子函数
3. [安全 - 重要] 未对用户输入做长度校验
```

### 6.3 终止条件

| 条件 | 动作 |
|------|------|
| 评分 ≥ 9.0 | ✅ 自动提交 + 合并 PR |
| 连续 3 轮评分未提高 | 🛑 停止迭代，标记为"需人工介入" |
| 达到最大轮次（默认 42） | 🛑 停止迭代，全量日志存档 |
| 连续 3 次测试失败 | 🛑 停止，记录"环境问题" |

## 七、落地建议

### 7.1 团队推行节奏

| 阶段 | 时长 | 重点 |
|------|------|------|
| **试点** | 第 1-2 周 | 选 2-3 个中等复杂度 Issue 跑完整流程 |
| **定型** | 第 3-4 周 | 根据反馈调整评分权重和阈值 |
| **推广** | 第 5-8 周 | 扩展到全团队的 PR 流程 |
| **优化** | 持续 | 积累评分数据，分析质量趋势 |

### 7.2 从人开始

千万不要一开始就全自动化。推荐路径：

1. **人工评审 + 评分工具辅助**：人负责评审，AI 提供评分参考
2. **AI 初审 + 人工复审**：AI 先过一遍，人看 AI 的结果
3. **全自动 + 人工抽检**：仅高分数段抽检，低分数段强制人工介入

### 7.3 需要什么样的基础设施

- [ ] delegate_task 支持 per-task 模型指定（目前 Hermes 需 context 声明）
- [ ] 评分结果持久化存储（用于趋势分析）
- [ ] 一个轻量级的 Webhook/通知机制（评分达标自动 PR）

## 八、总结

AutoResearch 方法论迁移到软件开发的核心洞察有三点：

1. **多模型交叉审核消灭盲区** — 不同模型有不同的弱项，交叉审核覆盖单个模型无法发现的问题
2. **量化评分驱动精准改进** — 5 维加权评分比二元判断多了一个信息维度，让 Agent 知道往哪个方向改
3. **Fast/Slow 分级兜底** — 不做一刀切，80% 的简单任务走快通道，20% 的复杂任务走慢通道反复打磨

**最终效果**：简单任务不浪费 token，复杂任务不牺牲质量。
