## 一、一个被你忽视的质量风险

### 📖 故事：沉默的降级

2026 年 2 月，一家 AI 客服 SaaS 公司的监控面板突然显示客户满意度下降 12%。没有部署、没有代码变更、没有配置改动——一切看起来正常。

工程师花了三天排查，最后发现问题出在模型提供商端：**OpenAI 在一次未公告的推理优化中，调整了 GPT-4o 的顶级 logit 采样参数。** 模型的回复风格变得更"直接"，不再进行礼貌性的确认和澄清。用户觉得机器人"变冷漠了"，于是给了差评。

没有版本号。没有 changelog。没有公告。你的"依赖"在你睡着的时候换了行为，而你直到用户骂上门才知道。

这不是故障——这是 AI 时代质量管理的底层困境。

### 这个问题比传统依赖更棘手

传统软件工程中，你的依赖链是明确的：

```
你的代码 → 库依赖（PyPI/npm）→ 基础设施（云服务）
```

每个层级都有成熟的治理手段：

| 层级 | 版本锁定 | 变更可见 | 可替代性 |
|------|:--------:|:--------:|:--------:|
| 代码库（PyPI） | ✅ `requirements.txt` 锁定版本 | ✅ 发布日志 | ✅ 可 fork |
| 云服务（AWS） | ✅ API 版本化 | ✅ SLA + status page | ⚠️ 迁移成本高但有方案 |
| **AI 模型** | ❌ 无法锁定版本 | ❌ 行为变更无公告 | ❌ 替代成本极高 |

模型层的依赖是**反软件工程**的：

- **没有语义版本号** — "GPT-4o" 这个字符串今天和昨天可以代表完全不同的行为
- **无 diff 可见性** — 改了参数、换了量化、更新了后处理——你永远不会被告知
- **bug 只能等修** — 你发现模型有特定缺陷，无法 patch，无法 downgrade，只能上报等待上游修复

### ⚠️ 反面案例：一次本该可预防的停摆

2025 年末，Anthropic 经历了一次大规模服务降级。某个推理集群的旧版 GPU 驱动不兼容新版部署脚本，导致部分请求返回空响应。这对 Anthropic 只是"4 小时 incident"。

但对依赖 Claude API 的团队：

- 一家法律 AI 公司的**合同审查自动流水线停摆 4 小时**——意味着当天积压的 200+ 份合同需要 8 个人加班到凌晨
- 一个客户支持 agent 系统**生成了 4 小时的空回复**——客户以为系统在工作，实际什么都没处理，直到客户投诉才发现
- 依赖该 agent 输出的下游系统**收到了 4 小时的空数据流**——触发了从未见过的级联故障

**每个团队都觉得"API 宕机这种小概率事件，遇上了再说"。但当它发生时，你没有备选方案。**

## 二、依赖链：比你想象的更收敛

### 你以为的选择是幻觉

市场上似乎有"多家"模型提供商可供选择。但把依赖链展开：

```
你的 Agent
  ├→ 模型 API（Anthropic / OpenAI / DeepSeek）
  │     ├→ 推理基础设施（GPU 集群）
  │     │     ├→ 云服务商（AWS / GCP / Azure）
  │     │     └→ GPU 供应商（NVIDIA）
  │     │           └→ 制造（台积电）
  │     └→ 基础模型权重（训练数据 + 算法）
  └→ 工具 API（Stripe / Notion / GitHub）
```

每一层都在收敛，而不是发散：

- **推理层** — 无论选哪家模型，跑在同一个 GPU 供应链上（H100 → B200 → Rubin）。Anthropic 挂了不是它一家的事，是它依赖的 GCP us-east1 区域的 GPU 集群出了问题
- **服务层** — API 的依赖模式高度同质化。OpenAI、Anthropic、Google 都走 REST API + SSE streaming，共享同一套底层网络基础设施
- **缺陷层** — 不同类型的模型会共享训练数据管道中的系统性偏差。当前三家主流模型在 2025 年都曾在同一个数学推理题目子集上集体翻车

### 链式故障的传播

传统软件的单点故障通常是"一条线断了，你绕路"。

AI 时代的链式故障是**层叠式的**：

```
GPU 驱动更新 → 推理延迟增加 → API 超时 → Agent 重试 → 排队加剧 → 雪崩
                        ↓
                  不是"慢一点"，是"整个链崩了"
```

更隐蔽的是**行为变更**的传播：

```
模型提供商调整了 alignment 策略
  → Agent 的回复变得更谨慎
    → 之前能处理的模糊请求现在频繁拒答
      → 下游审计系统标记了大量"异常拒绝"
        → 人工审核团队被无效告警淹没 → 真正的问题被淹没在噪音里
```

## 三、模型缺陷：你无法自己修的 bug

### 模型不是库

传统依赖出 bug 的标准路径：

```
发现 bug → 去 GitHub issues 搜 → 等 patch → `pip install --upgrade`
```

模型缺陷的路径：

```
发现特殊字符导致模型死循环 → 搜遍所有论坛确认不是自己配错
  → 确认是模型本身的 tokenizer bug
    → 向提供商提交工单
      → 回复"已知问题，正在修复"
        → 等待（时间不确定，可能是一周，可能是三个月）
          → 在等到的这段时间里，你只能建临时防御层
```

我们自己在 Hermes 项目上就踩过这个坑：**DS V4 对特定 Unicode 序列（U+200D 等零宽字符）的处理会导致模型无限循环。** 这不是 API 的问题，是模型本身 tokenizer 的缺陷——不管跑在本地还是云端都会重现。唯一的解决方案是上游发新 checkpoint。

### 你面对的几种"修不了"的 Bug

| 类型 | 表现 | 你能做什么 | 上游修复周期 |
|------|------|-----------|:----------:|
| **Tokenizer 缺陷** | 特殊字符触发死循环/重复 | 输入 sanitization | 周~月 |
| **Alignment 漂移** | 模型变得过度谨慎或过度冒险 | 调整 system prompt | 不确定（策略变更） |
| **量化损失** | 推理优化后特定场景准确率下降 | 切换模型版本 | 通常不会回滚 |
| **上下文窗口退化** | 长上下文下越到末尾效果越差 | 分块/窗口滑动 | 月~季 |
| **Function Calling 偏差** | 工具调用行为不一致 | 多模型交叉验证 | 次~月 |
| **Benchmark 过度优化** | 常见场景表现好但边缘场景差 | 持续测自己的场景 | 版本迭代 |

### ⚠️ 反面案例：U+200D 的蝴蝶效应

2026 年 4 月，我们的 Telegram 频道监控 cron job 连续失败。错误信息是："Blocked: prompt contains invisible unicode U+200D (possible injection)。"

追查后发现：
1. Telegram 消息中含有零宽连字（U+200D），是 emoji 序列的一部分
2. 脚本把消息原文输出到 agent prompt
3. 安全模块检测到不可见字符，判定为注入攻击，阻止了执行
4. 而这个 U+200D 字符在被之前某个模型版本处理时**曾经触发过无限循环**

两层事故叠加：

```
用户发 emoji → 消息含 U+200D → 脚本输出到 prompt → 重复注入检测
                                                          ↓
                                              被模型历史缺陷牵连
```

最终修复不是等模型提供商修，而是在脚本中加了一层 sanitization——**针对"你也不知道这个模型有什么问题，但先堵上"的防御性过滤**。

这将成为 AI 时代的常态：**在模型质量达到可靠之前，你的系统每一层都要假设模型有已知和未知的缺陷。**

## 四、API 作为基础设施：SLA 的盲区

### 你签了什么样的合同

回顾你使用的模型 API 的 ToS，很可能包含这样的条款：

> "服务可能随时修改或终止，恕不另行通知。"
> "我们对因服务变更或中断造成的损失不承担责任。"
> "SLA 额度仅以积分形式返还，不涉及现金赔偿。"

这不是"服务合同"，这是**单方免责声明**。

对比一下，云服务商（AWS/Azure/GCP）的 SLA：

| 维度 | 云服务（AWS EC2） | 模型 API（行业现状） |
|------|:-----------------:|:------------------:|
| 可用性承诺 | 99.99%（四九） | 99.9%+（但赔付极低） |
| 行为变更 | API 版本化，V1/V2 共存 | 无版本锁定，静默变更 |
| 宕机赔付 | Service Credit（最高 100%月费） | 极低倍数 |
| 功能 deprecation | 提前 12 个月公告 | 无明确承诺 |
| 第三方审计 | SOC2/ISO27001 | 部分有 |

问题不在于"模型 API 的 SLA 不好"，而在于**依赖性远高于云服务，但服务品质承诺远低于云服务。** 你可以在 AWS 上跑多 AZ 部署来对冲单 AZ 故障；但你无法"多模型部署"来对冲模型行为变更——因为每个模型的 prompt 是定制的，行为迁移成本远高于云服务的跨 AZ 迁移。

### 你应该在合同里要求的

如果你的业务依赖模型 API（不只是"试试看"的阶段），以下条款应该写入采购合同或服务协议：

```
1. 变更公告义务
   - 任何影响输出行为的变更（模型权重更新、推理优化、采样参数调整），
     至少提前 14-30 天公告
   - 附带行为变更的详细说明（diff 形式）

2. 重大缺陷披露
   - 已知且有实际影响的模型缺陷，在 status page 上公开
   - 包括：触发条件、影响范围、预计修复时间线

3. SLA 追溯
   - 持续降级（latency 翻倍、错误率超标 2× 超过 30 分钟）
     应纳入 SLA 赔付范围
   - 不限于"完全不可用"，也包括"服务质量重大降级"

4. 定价保护期
   - 价格变更至少 30 天公告
   - 重大变更（指价格翻倍或更换计费模型）用户在公告期内有权无惩罚退出
```

**现实是**：作为个人开发者或小团队，这些条款几乎不可能谈下来。但如果你在做 B2B 产品或关键业务系统，这些条款应当成为采购流程的标准部分——就像你不会签一个"随时可能改行为、断了不赔"的云服务合同一样。

## 五、你能做的架构级防御

"等上游修"不是选项。以下是 2026 年可行的四个架构层防御手段。

### 第一道防线：模型抽象层

```
不是：app → OpenAI
而是：app → Gateway → {OpenAI, Anthropic, DeepSeek, 本地}
                      ↕
                  统一接口适配层
```

Hermes 的 fallback 机制就是这种思路，但要更进一步——不是"主模型失败才切备用"，而是**每个请求可以按策略选择模型**：

- 简单任务 → DeepSeek V4（成本低）
- 代码生成 → Claude Sonnet（代码质量高）
- 长文档分析 → Gemini 2.5（长上下文强）
- 高价值请求 → 并行发两个模型，对比结果

关键不是"有多个选项"，而是**所有选项共享同一套 prompt 模板和输出校验逻辑**。

### 第二道防线：输出校验层

模型的输出需要经过校验才能往下流：

```
模型输出 → 格式校验 → Schema 校验 → 语义合理性校验
         → 循环检测（模型卡住时自动截断/重试）
         → 异常内容过滤（如不可见 Unicode）
         → 与上次输出的 diff（检测静默行为变化）
         ↓
      如果校验不通过 → fallback 到备用模型
```

这块类似传统 API gateways 做的事，但针对模型特性做了扩展——**校验的是语义层面的正确性，不只是 JSON Schema 合规性。**

### 第三道防线：持续基准测试

别信厂商的 benchmark。在你的真实数据和任务上持续跑：

```
每周自动化流程：
  1. 从生产环境采样 100 条真实请求（脱敏后）
  2. 发给所有候选模型
  3. 用 LLM-as-judge 或人工评分比较结果
  4. 记录评分趋势线
  5. 如果某个模型连续两周评分下降 → 自动标记告警
```

这套流程让你在模型静默降质时**3 天内知道，而不是等到用户投诉三周后**。

### 第四道防线（逃生舱）：最小可行能力层

有一个独立于所有 API 的能力层，保证"不死透"：

```
当 API 全挂时 → 降级到本地模型
  输入 → 本地模型（Qwen 3.6 35B / Gemma 4 26B）
       → 核心功能可用，质量降级但业务不停摆
       → 所有输出标记"降级模式"，人知道要更仔细地审
```

这不是"本地替代 API"，这是**业务连续性设计**——跟飞机上的备用仪表盘一个逻辑。你可能 99% 的时间用不上，但它决定了 1% 的极端情况发生时，你的系统是"还能用"还是"彻底白屏"。

### 📖 真实案例：Gemma 4 作为逃生舱

我们自己在 T2 机器（AMD HX370 / 96GB 内存）上跑了 LM Studio，部署了 **Gemma 4 E4B**（26B MoE，约 4B 活跃参数）和 **Gemma 4 26B a4b** 作为本地逃生舱。配置如下：

```
本地推理节点
├── 硬件: HX370 + 96GB RAM（消费级）
├── 服务: LM Studio API  (:8001)
├── 模型: Gemma 4 E4B（日常降级用）
│           Peach-2.0-9B-8k-RP（聊天/角色）
│           Gemma 4 26B a4b（复杂推理兜底）
├── 路由: socat 代理桥接（7891→7890）
└── 接入: Hermes 配置 custom_providers
         → fallback_providers 自动切换
```

当 API 正常时，日常走 DeepSeek V4。当 API 不可用或异常（高延迟、重复响应、空响应）时，Hermes 的 fallback 机制自动切到本地 LM Studio 的 Gemma 4。这是生产环境验证过的**零配置切换**——不需要改代码、不需要重启服务，只是响应质量降级。

这个配置有两个关键点值得注意：

1. **消费级硬件就够了** — 26B MoE 在 96GB 内存上跑推理足够覆盖翻译、摘要、分类、简单代码生成。不是替代 API 的顶级推理能力，而是**保你不死透**。
2. **延迟优势** — 本地推理的延迟是稳定的（不受远端负载影响），在 API 高负载时的 latency 可能比远程更优。降级模式的核心其实是"可预测的慢"。

### 降级模式的真实触发场景

根据实际运行经验，以下场景会触发降级：

| 触发条件 | 表现 | 本地模型表现 | 恢复策略 |
|---------|------|:----------:|:--------:|
| API 宕机/超时 | 连续 3 次请求失败 | ✅ 可用但质量降级 | API 恢复后自动切回 |
| API 延迟飙升 | 响应时间 > 30 秒 | ✅ 延迟稳定（~5 秒/请求） | API 延迟恢复后自动切回 |
| 模型输出异常 | 空响应/无限循环/乱码 | ✅ 正常输出 | 手动阈值回归 |
| 余额/配额耗尽 | 返回 402/429 | ✅ 不受影响 | 充值后自动恢复 |
| 网络中断 | 无法连接远端 | ✅ 完全离线工作 | 网络恢复后自动同步 |

### 建立逃生舱的检查清单

如果你也想搭建本地逃生舱，以下是关键步骤：

```
☐ 硬件确认：内存 ≥ 32GB（跑 7B 量级）/ ≥ 64GB（跑 MoE 26B 量级）
☐ 推理引擎：LM Studio / llama.cpp / Ollama 三选一
☐ 模型下载：确认你需要的模型已下载到本地（不要等宕机了才下，几十 GB 下不完）
☐ API 兼容：确认本地推理引擎暴露 OpenAI 兼容 API
☐ 路由配置：在 agent 框架中配置 fallback provider
☐ 定期测试：每月手动触发一次降级模式，确保逃生舱能正常工作（很多人配了，但从来没试过，到了真用的时候发现模型没下全）
☐ 质量基线：在你最常用的 3-5 个任务类型上跑一次本地模型 vs API 模型的对比评分，知道降级到多少分
```

### ⚠️ 反面：逃生舱不是万能的

本地模型的局限性也必须坦诚：

```
❌ 本地模型跑不动复杂 agent 编排（多步推理+工具调用）
   → 降级模式下，agent 仍可用，但从"自动化执行"降级为"辅助生成"

❌ 本地模型的多模态能力弱
   → 涉及图像理解的任务在降级模式下直接不可用

❌ 量化损失不可控
   → 为了跑在消费级硬件上，模型量化为 4-bit/6-bit，特定任务的质量折损无法精确预测

❌ 模型更新靠上游
   → 本地模型同样有 tokenizer bug、alignment 问题——只是你能原地不动，不用等 API 恢复
```

**逃生舱的意义不是"一样好"，而是"一样有"**。当 API 挂了 4 小时，你用 Gemma 4 跑了 3 小时 45 分钟的工作——这些工作虽然质量打了折扣，但业务没有完全停摆。这才是逃生舱的核心价值。

## 六、关于监管：理性期待，积极自保

### 现状：监管关注了错误的事

2026 年全球 AI 监管（EU AI Act、中国生成式 AI 管理办法、美国行政令）主要聚焦在：

| 监管焦点 | 是否重要 | 和你最相关的问题 |
|---------|:-------:|:--------------:|
| 模型输出有害内容 | ✅ 重要 | **模型稳定性** |
| 深度伪造/虚假信息 | ✅ 重要 | **API 可用性** |
| 数据隐私/训练数据版权 | ✅ 重要 | **行为变更可见性** |
| 算法歧视/偏见 | ✅ 重要 | **定价透明** |
| **模型缺陷透明性** | **❌ 没人管** | **你在被 bug 坑时无从追责** |

监管的空白地带恰好是你的最大痛点：**API 作为一个关键基础设施，没有基础设施该有的责任约束。**

### 应该推动的监管方向

好的监管不是告诉你"该用哪个模型"，而是确保你**知情且有选择**：

```
1. 已知缺陷的强制披露
   类似 CVE 的模型缺陷数据库。
   被外部确认的、有实际影响的问题，厂商必须在合理时间内公开状态。

2. API 的"基础设施"身份认定
   当模型 API 被用于高影响场景（医疗、金融、关键基础设施），
   提供商必须满足基础设施级 SLA，包括变更公告义务和故障透明报告。

3. 变更公告期
   行为变更、价格变更需要合理的告示期（30-60 天），
   用户有权在重大变更后无惩罚退出。
```

### ⚠️ 反面：监管做歪更可怕

监管方向如果错了，反而会加剧你面临的问题：

1. **牌照化 = 固化巨头** — 合规成本太高，DeepSeek、Mistral 这种小公司承担不起。结果不是更多选择，而是**更少选择**。
2. **过度监管扼杀开源** — 如果开源模型也要遵守同样规则，Meta 可能不敢放 Llama，DeepSeek 不敢开源新版本。你失去了本地部署作为逃生舱的可能性。
3. **政治化 = 碎片化** — EU 一套、中国一套、美国各州不同。全球化产品要对接三套合规要求。

### 你现在的行动清单

监管是长跑。眼前你能做的是：

```
☐ 短期（这周能动手）
   → 确认你的系统架构层是否支持多模型切换
   → 在输入/输出两端加上基础防御（特殊字符过滤、格式校验、循环检测）
   → 建立一个简单的模型表现对比表（你的真实任务，不同模型的表现）

☐ 中期（1-3 个月）
   → 画真实依赖链（本章动手环节）
   → 建立持续的模型评估流程（每周自动跑基准测试）
   → 评估搭建本地逃生舱的成本与收益
   → 如果是 B2B 产品，在采购合同里加入模型 API 变更公告条款

☐ 长期（6-12 个月）
   → 关注你所在地区的监管动态，但不要等监管来解决你的架构问题
   → 保持"多模型"架构习惯，不要让自己陷入"不用这家就完了"的境地
   → 持续投入本地推理能力——不是为了替代 API，而是为了你的业务有个底线
```

## 七、附录：模型提供商风险尽职调查清单

选择模型 API 提供商时，可以用下面的清单逐项评估。每项评分 1-5 分，总分 100 分，**低于 60 分不建议作为主力依赖**。

### A. 基础能力（30 分）

| # | 评估项 | 评分标准 | 得分 |
|:-:|--------|---------|:---:|
| 1 | **可用性 SLA** | 99.9%=1分, 99.95%=3分, 99.99%=5分 | __/5 |
| 2 | **SLA 赔付力度** | 仅积分=1分, 月费10%=3分, 月费50%+=5分 | __/5 |
| 3 | **连续宕机记录** | 近6个月有2次+严重事故=1分, 1次=3分, 0次=5分 | __/5 |
| 4 | **多 region 部署** | 单region=1分, 多region可选=3分, 自动跨region=5分 | __/5 |
| 5 | **API 版本化** | 无版本概念=1分, 有版本号但无共存=3分, V1/V2共存=5分 | __/5 |
| 6 | **延迟 P99** | >5秒=1分, 2-5秒=3分, <2秒=5分 | __/5 |

### B. 变更管理（20 分）

| # | 评估项 | 评分标准 | 得分 |
|:-:|--------|---------|:---:|
| 7 | **行为变更公告** | 无公告=1分, 事后公告=3分, 事前公告+diff=5分 | __/5 |
| 8 | **模型 freeze 能力** | 无=1分, 可指定快照版本=3分, 有stable/edge渠道=5分 | __/5 |
| 9 | **deprecation 周期** | 无通知=1分, 30天=3分, 90天+=5分 | __/5 |
| 10 | **定价稳定性** | 近6个月调价2次+=1分, 1次=3分, 未变=5分 | __/5 |

### C. 透明性（20 分）

| # | 评估项 | 评分标准 | 得分 |
|:-:|--------|---------|:---:|
| 11 | **status page 质量** | 无=1分, 有但延迟=3分, 实时+root cause=5分 | __/5 |
| 12 | **已知缺陷披露** | 不披露=1分, 部分披露=3分, 公开bug tracker=5分 | __/5 |
| 13 | **模型卡完整性** | 只有好的=1分, 有benchmark=3分, 有已知限制=5分 | __/5 |
| 14 | **独立安全审计** | 无=1分, SOC2/ISO27001=3分, 年度第三方审计报告公开=5分 | __/5 |

### D. 可替代性（15 分）

| # | 评估项 | 评分标准 | 得分 |
|:-:|--------|---------|:---:|
| 15 | **API 兼容性** | 私有协议=1分, 部分兼容OpenAI=3分, 完全兼容=5分 | __/5 |
| 16 | **数据可迁移性** | 无导出=1分, 可导出=3分, API批量导出=5分 | __/5 |
| 17 | **模型权重可获取** | 闭源不可得=1分, 开源但需审批=3分, 完全开源可下载=5分 | __/5 |

### E. 社区与生态（15 分）

| # | 评估项 | 评分标准 | 得分 |
|:-:|--------|---------|:---:|
| 18 | **文档质量** | 不全/过时=1分, 可查=3分, 完整+示例+迁移指南=5分 | __/5 |
| 19 | **社区活跃度** | 无人维护=1分, 论坛有回应=3分, 官方人员活跃=5分 | __/5 |
| 20 | **企业支持渠道** | 仅社区=1分, 工单=3分, 专属TAM=5分 | __/5 |

**评分解读：**

```
90-100 🟢 主力依赖可选
70-89  🟡 可使用，但需准备备用方案
60-69  🟠 慎用，仅限非关键场景
<60    🔴 不建议作为核心依赖
```

即使某个提供商得分很高，也**不要成为唯一依赖**。最佳实践是选得分最高的两个模型提供商，各自承载一部分流量，并且都能在短时间内承担全部流量。

## 八、附录：合同条款模板

以下条款模板适用于 B2B 采购场景。如果你们公司采购模型 API，可以参照这些条款写入服务合同。

### 条款 1：行为变更通知

**适用场景：** 模型的行为可能随更新而变化，你需要在变更前得到通知以便测试。

> **建议表述：**
> "服务方对模型进行任何可能影响输出内容、格式、风格或准确性的变更（包括但不限于基础模型更新、推理优化、量化方式调整、采样参数默认值变更），应至少提前 **[30/60]** 日以书面形式通知采购方。
>
> 通知内容应包括：
> (a) 变更的详细技术说明；
> (b) 已知的预期行为差异（以变更前后对比样本形式）；
> (c) 变更的生效日期。
>
> 采购方有权在收到通知后 **[15]** 日内要求对变更进行测试部署。如测试发现变更对采购方业务产生重大不利影响，采购方有权要求延迟变更生效时间或选择继续使用变更前版本不超过 **[90]** 日。"

### 条款 2：重大缺陷披露义务

**适用场景：** 模型存在已知的、有实际影响的 bug，你需要知道它存在——而不是等用户投诉才发现。

> **建议表述：**
> "服务方应在知悉或应当知悉模型存在重大缺陷（指可能导致输出内容在特定输入条件下不正确、不完整、包含有害内容或无限循环的缺陷）后 **[48]** 小时内，在公开 status page 上披露以下信息：
>
> (a) 缺陷的描述和触发条件；
> (b) 已确认受影响的使用场景；
> (c) 预计修复时间线或临时缓解措施。
>
> 若缺陷存在时间已超过 **[14]** 日且未修复，采购方有权在通知后终止受影响服务的使用，并获得按未使用期限比例退还的服务费用。"

### 条款 3：服务降级 SLA

**适用场景：** 比"完全宕机"更常见的是"服务质量降级"——响应变慢、错误率升高，但服务没停。

> **建议表述：**
> "除服务不可用外，以下情形应被视为服务质量降级事件：
>
> (a) API 响应时间 P95 超过正常基线值的 **[3]** 倍且持续超过 **[15]** 分钟；
> (b) API 错误率（非 200/非正常响应）超过 **[5]**% 且持续超过 **[15]** 分钟；
> (c) 有效输出率（非空响应、非重复响应、非截断响应）低于 **[95]**%。
>
> 上述降级事件每次按 **[0.5]** 次不可用事件计入 SLA 计算。月度累计降级时间超过 **[60]** 分钟，采购方有权获得当月服务费 **[10]**% 的 credit。"

### 条款 4：价格变更保护

**适用场景：** 模型 API 定价随时可能变，你的预算需要可预测。

> **建议表述：**
> "服务方调整服务价格或计费模式，应至少提前 **[30]** 日书面通知采购方。
>
> 若价格上调幅度超过 **[15]**%，或计费模式发生根本性变更（如从按 token 计费改为按请求计费），采购方有权在 **[30]** 日内无惩罚终止服务合同，并获得按未使用期限比例退还的预付费用。
>
> 价格冻结期：服务方承诺自合同生效日起 **[6/12]** 个月内不调整约定费率。"

### 条款 5：数据所有权与可迁移性

**适用场景：** 你的数据不应该因为换了模型服务商就被锁死。

> **建议表述：**
> "采购方在使用服务过程中产生的所有输入数据、输出数据和中间数据，其所有权归采购方所有。
>
> 服务方应在合同终止或采购方要求时，在 **[7]** 日内以通用格式（JSON/CSV）提供采购方全部数据的导出文件。
>
> 服务方不得以数据格式、接口兼容性或技术限制为由，阻碍采购方将数据迁移至第三方服务。服务方应提供 API 接口文档以支持采购方进行数据迁移。"

### 条款 6：责任上限（对采购方有利的改良版）

**适用场景：** 标准 ToS 中的责任上限通常极低（"只退当月费用"），对依赖模型 API 做核心业务的团队不够。

> **建议表述：**
> "因服务方违反 SLA 或本合同项下义务，导致采购方遭受直接损失的，服务方的累计赔偿上限应不低于：
> (a) 过去 **[3/6/12]** 个月采购方向服务方支付的服务费总额；或
> (b) **[具体金额，建议根据业务影响评估]**，
>
> 以较高者为准。
>
> 因服务方故意或重大过失、或违反数据安全义务所导致的损失，不适用前述责任上限。"

### 如何使用这些条款

```
🔴 如果你是个人开发者
   → 几乎无法谈判。建议关注服务公开的 ToS，优先选择 ToS 中对开发者更友好的提供商。
      DeepSeek、Mistral 等中小厂商的 ToS 通常比大厂更灵活。

🟡 如果你在创业公司（1-20 人）
   → 选择年付/承诺消费模式时，可以附带要求部分条款。
      通常月份越多，谈判空间越大。优先争取条款 1（变更通知）和条款 4（价格保护）。

🟢 如果你在中大型企业（有法务团队）
   → 以上条款可以作为采购流程的基线。
      建议法务根据企业实际风险管理策略调整金额和期限，并特别注意管辖法律和争议解决条款。
```

## 本章小结

> AI 时代的供应链质量管理，核心矛盾在于：
> **对模型的依赖强度远超传统软件依赖，但模型的行为可控性远低于传统软件依赖。**

这不是一个能被"选个更可靠的提供商"解决的问题——每层都在收敛，选择空间被层层压缩。

有效的应对策略不是追求"解决风险"（它无法被解决），而是：

1. **知情** — 知道你的依赖链有多长、每个节点有多脆弱
2. **对冲** — 模型抽象层 + 输出校验层 + 持续基准测试
3. **保底** — 本地逃生舱 + 降级设计 + 合同的底线性保护条款

**没有 "set and forget"。没有 "rely on one vendor"。没有 "they'll fix it for us"。** 这是一个需要持续管理的风险敞口。承认这一点，然后动手建防御层。
