> **一句话摘要**：本文详解 Hermes Agent 现有的 subagent-driven-development 方案——这是理解 AutoResearch 改进方向的前提，也是团队日常开发中 80% 场景的默认路径。

---

## 一、方案定位

Hermes 的 `subagent-driven-development` skill（以下简称 SDD）是 Hermes Agent 内置的**自动化开发执行引擎**。

它解决的问题：当你有一个明确的实现计划后，如何可靠地将计划变成代码——自动完成、自动审核、自动合并。

**它不是** AutoResearch 的替代品，而是 **AutoResearch 改进的基线**。

---

## 二、核心架构

SDD 的核心是一个**三阶段四步走**的流水线：

```
[Plan 输入]
    │
    ▼
┌─────────────────────────────────────────┐
│          阶段一：单任务执行                │
│                                         │
│  Step 1: 实施者 Subagent                 │
│    • 读 task 描述 → 写测试 → 实现        │
│    • TDD 循环（RED→GREEN→REFACTOR）      │
│    • 输出：代码 + 通过的测试              │
└──────────────┬──────────────────────────┘
               ▼
┌─────────────────────────────────────────┐
│          阶段二：双阶段审核                │
│                                         │
│  Step 2: 规范审核（Spec Review）          │
│    • 检查：需求是否全部实现？              │
│    • 检查：有无 scope creep？             │
│    • 输出：PASS 或 差距列表               │
│                                         │
│  Step 3: 质量审核（Quality Review）       │
│    • 代码风格、错误处理、测试覆盖           │
│    • 输出：APPROVED 或 问题列表            │
└──────────────┬──────────────────────────┘
               ▼
┌─────────────────────────────────────────┐
│          阶段三：全局整合                  │
│                                         │
│  Step 4: 集成审核（Final Review）         │
│    • 检查跨任务一致性                     │
│    • 完整测试套件                         │
│    • Ready for merge                     │
└─────────────────────────────────────────┘
```

### 2.1 每个任务的生命周期

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 实施 Subagent│ ──→ │ 规范审核       │ ──→ │ 质量审核       │
│ （写代码）    │     │ （查需求命中） │     │ （查代码质量） │
└──────────────┘     └──────────────┘     └──────────────┘
       │                    │                     │
       │ ← ─ ─ ─ 修复 ─ ─ ─┤ ← ─ ─ ─ 修复 ─ ─ ─┤
       │                    │                     │
       ▼                    ▼                     ▼
   下一任务               PASS                  APPROVED
```

**关键规则**：
- 规范审核**必须先于**质量审核
- 发现问题后回退给实施者修复，修复后**重新审核**
- 不跳过任何审核步骤

## 三、核心设计原则

### 3.1 原则一：每个任务一个干净的 Subagent

```
❌ 错误方式：一个 Subagent 做所有任务
    Subagent A: 做任务1→任务2→任务3→任务4
    → context 里越堆越多，Agent 开始混淆

✅ 正确方式：每个任务一个新 Subagent
    Subagent A: 任务1 → 销毁
    Subagent B: 任务2 → 销毁
    Subagent C: 任务3 → 销毁
    → 每个 Subagent 只看到自己需要的信息
```

**为什么这么做**？
- 防止 context 污染（Agent 不会被上一个任务的信息干扰）
- 每个 Subagent 的 context 长度可控
- 单个 Subagent 失败不影响其他任务

### 3.2 原则二：两阶段审核分离关注点

```python
# 阶段 1：规范审核（查需求）
审核员查看：
- 任务描述中的每一条需求是否都实现了？
- 有没有实现需求之外的东西（scope creep）？
- 文件路径、函数签名、接口定义是否符合预期？

# 阶段 2：质量审核（查实现）
审核员查看：
- 代码风格是否符合项目规范？
- 错误处理是否完善？
- 测试覆盖是否充足？
- 有无明显 bug？
```

**分离的好处**：一个 Subagent 专门查"做对了没有"，另一个查"做得好不好"。专注力不同，发现的盲区也不同。

### 3.3 原则三：审核反馈必须闭环

```
审核发现问题 → 实施者修复 → 审核员再审 → APPROVED
                    ↑                    │
                    └── 仍有问题？──────┘
```

不允许"发现问题就放过"或"下次再改"。

## 四、与 TDD 的集成

SDD 在每个实施步骤中内置了 TDD 流程：

```python
# 实施者的工作指令
1. 创建测试文件 tests/models/test_user.py
2. 写入测试用例（预期的行为和接口）
3. 运行: pytest tests/models/test_user.py -v
   → 验证: 测试应该 FAIL（RED 阶段）
4. 创建实现文件 src/models/user.py
5. 写入最简实现让测试通过
6. 运行: pytest tests/models/test_user.py -v
   → 验证: 测试应该 PASS（GREEN 阶段）
7. 重构（如果有需要）
8. 运行: pytest tests/ -q
   → 验证: 全部测试通过，无回归
```

## 五、方案优势与局限

### 5.1 优势

| 维度 | 表现 |
|------|------|
| **速度** | 简单任务一次通过，2-5 分钟 |
| **可靠性** | TDD 确保代码有测试保护 |
| **可审计** | 每个步骤都有日志 |
| **成本** | 每个任务仅 1× delegate_task token 消耗 |

### 5.2 局限（也是 AutoResearch 改进的方向）

| 局限 | 表现 | AutoResearch 的改进 |
|------|------|-------------------|
| **单模型盲区** | 实现和审核用同一模型，发现不了自己的盲区 | **多模型交叉**（Codex↔Claude） |
| **二元判断** | 审核只有通过/不通过，缺乏梯度信息 | **5 维量化评分**（0-10） |
| **无迭代** | 一次性完成，不重试 | **反馈驱动迭代**（直至达标） |
| **无共同宪章** | 每个 Subagent 的规则写在各自的 context 中 | **program.md**（统一宪法） |
| **无评分历史** | 审核结果不持久化 | **完整评分历史**（趋势分析） |

## 六、典型场景示例

### 6.1 简单任务（Fast Path 默认）

```python
# 任务：给 User model 加一个 phone 字段
delegate_task(
    goal="在 User model 中增加 phone 字段",
    context="""
    修改 src/models/user.py:
    - 添加 phone: str = "" 字段
    - 在 __init__ 中初始化
    - 更新测试
    
    按 TDD 流程执行
    """
)
# → 一次通过，3 分钟完成
```

### 6.2 复杂任务（应升级到 Slow Path）

```python
# 任务：实现用户认证系统
# → 涉及多个文件、安全处理、token 管理
# → SDD 可能一次不过，审核发现问题
# → 此时应自动升级到 AutoResearch Slow Path
```

## 七、如何判断"什么时候该升级"

基于 SDD 的运行数据，总结以下升级触发条件：

| 条件 | 典型表现 | 行动 |
|------|---------|------|
| 审核发现**架构级问题** | "这个设计可能需要重新考虑" | 立即升级 |
| 审核者列出了 **3+ 个严重问题** | 测试覆盖差 + 安全隐患 + 代码混乱 | 立即升级 |
| **跨 3 个以上文件**的改动 | 涉及多个模块 | 直接走 Slow Path |
| 涉及**支付/安全/用户数据** | 任务 tag 含 security | 直接走 Slow Path |
| 第一次修复后审核**仍有严重问题** | 修复不彻底 | 升级 |

## 八、团队落地指引

### 8.1 开发人员需要知道什么？

**最少必要知识**——以下三条就够了：

1. **写 plan** — 任务拆得越小，SDD 执行越稳。每个任务 2-5 分钟的工作量
2. **读审核结果** — 审核不通过不丢脸，修复就好。找到问题比没发现好
3. **标记复杂任务** — 如果直觉告诉你"这个可能有点麻烦"，在 plan 里加个 `tags: [complex]`

### 8.2 技术 Leader 需要关注什么？

- **升级率**：Fast→Slow 升级率是多少？过高说明 plan 拆得不好，过低说明 Fast Path 太松
- **审核问题类型**：最常见的是测试不够，还是代码质量，还是安全遗漏？
- **Subagent 成功率**：实施者一次通过的比率，低于 60% 说明 context 中缺少关键信息

---

## 九、总结

Hermes Subagent-Driven-Development 是一套**可靠但简单**的自动化开发方案：

> 每个任务一个干净的 Subagent → TDD 实现 → 两阶段审核 → 完成

它覆盖了日常开发 80% 的场景。对于剩下的 20% 复杂任务，它提供充分的**判断依据**来决定是否升级到 AutoResearch 的 Slow Path。

**SDD 不是替代品，而是基础层——AutoResearch 是建立在这个基础之上的增强层。**
